#+title: Customizing the Template for Your Project

This guide explains how to adapt this template for your specific project.

* Quick Rename Guide

** 1. Update Project Names

Replace ~rust-template~ with your project name throughout:

#+begin_src sh :exports code
# Example: renaming to "my-app"
PROJECT_NAME="my-app"

# Update Rust code and configs
find . -type f \( -name "*.toml" -o -name "*.rs" -o -name "*.org" \) \
  -not -path "*/target/*" \
  -exec sed -i '' "s/rust-template/${PROJECT_NAME}/g" {} +

find . -type f \( -name "*.toml" -o -name "*.rs" -o -name "*.org" \) \
  -not -path "*/target/*" \
  -exec sed -i '' "s/rust_template/${PROJECT_NAME//-/_}/g" {} +
#+end_src

** 2. Update Cargo.toml Metadata

Edit the workspace ~Cargo.toml~:

#+begin_src toml :exports code
[workspace.package]
version = "0.1.0"
edition = "2021"
authors = ["Your Name <your.email@example.com>"]
license = "MIT OR Apache-2.0"  # or your preferred license
repository = "https://github.com/yourusername/your-project"
#+end_src

** 3. Update flake.nix Workspace Crates

Edit ~flake.nix~ and update the ~workspaceCrates~ section:

#+begin_src nix :exports code
workspaceCrates = {
  # CLI application
  cli = {
    name = "my-app-cli";           # Cargo package name
    binary = "my-app-cli";         # Binary executable name
    description = "CLI application";
  };

  # Server process
  server = {
    name = "my-app-server";
    binary = "my-app-server";
    description = "Server process";
  };

  # Add more crates as needed
  # worker = {
  #   name = "my-app-worker";
  #   binary = "my-app-worker";
  #   description = "Background worker";
  # };
};
#+end_src

The ~workspaceCrates~ configuration in ~flake.nix~ is the *single source of
truth* for:
- Nix package generation (when uncommented)
- App generation (~nix run .#cli~, ~nix run .#server~)
- Overlay generation for NixOS modules
- Consistent naming across the flake

This prevents repetitive boilerplate and makes it easy to add/remove workspace
members.

** 4. Rename Directories

#+begin_src sh :exports code
# If you want to rename the crate directories too:
mv crates/cli "crates/${PROJECT_NAME}-cli"
mv crates/server "crates/${PROJECT_NAME}-server"
mv crates/lib "crates/${PROJECT_NAME}-lib"

# Update Cargo.toml workspace members accordingly
#+end_src

* Adding New Workspace Crates

** Example: Adding a Background Worker

1. Create the crate:

#+begin_src sh :exports code
cargo new --bin crates/worker
#+end_src

2. Add to workspace ~Cargo.toml~:

#+begin_src toml :exports code
[workspace]
members = [
    "crates/cli",
    "crates/lib",
    "crates/server",
    "crates/worker",  # New!
]
#+end_src

3. Update package ~Cargo.toml~:

#+begin_src toml :exports code
# crates/worker/Cargo.toml
[package]
name = "my-app-worker"
version.workspace = true
edition.workspace = true
authors.workspace = true
license.workspace = true
repository.workspace = true

[[bin]]
name = "my-app-worker"
path = "src/main.rs"

[dependencies]
my-app-lib = { workspace = true }
# ... other dependencies
#+end_src

4. Add to ~flake.nix~ workspaceCrates:

#+begin_src nix :exports code
workspaceCrates = {
  cli = { ... };
  server = { ... };

  # Add the new worker
  worker = {
    name = "my-app-worker";
    binary = "my-app-worker";
    description = "Background worker";
  };
};
#+end_src

5. Implement with the same patterns:
   - Create ~config.rs~ with staged configuration types
   - Use semantic error types with ~thiserror~
   - Add integration tests

* Removing Crates You Don't Need

** Example: Removing the Server Crate

1. Remove from workspace:

#+begin_src toml :exports code
# Cargo.toml
[workspace]
members = [
    "crates/cli",
    "crates/lib",
    # "crates/server",  # Commented out or removed
]
#+end_src

2. Remove from ~flake.nix~:

#+begin_src nix :exports code
workspaceCrates = {
  cli = {
    name = "my-app-cli";
    binary = "my-app-cli";
    description = "CLI application";
  };

  # server = { ... };  # Removed
};
#+end_src

3. Delete the directory:

#+begin_src sh :exports code
rm -rf crates/server
#+end_src

* Customizing Configuration

** Adding Application-Specific Config

Each application (~cli~, ~server~) has its own ~config.rs~.  Add your settings
there:

#+begin_src rustic :results none :exports code
// In crates/cli/src/config.rs

#[derive(Debug)]
pub struct Config {
    pub log_level: LogLevel,
    pub log_format: LogFormat,

    // Add your application-specific config
    pub database_url: String,
    pub api_key: Option<String>,
    pub max_retries: u32,
}

impl Config {
    pub fn from_cli_and_file(cli: CliRaw) -> Result<Self, ConfigError> {
        // Merge CLI args, config file, and defaults
        // Validate and construct final Config
    }
}
#+end_src

Remember to update:
- ~CliRaw~ (for command-line arguments)
- ~ConfigFileRaw~ (for file-based config)
- ~from_cli_and_file()~ merge logic
- Semantic error types if validation fails

* Enabling Nix Package Building

When you're ready to build Nix packages, uncomment the packages section in
~flake.nix~:

#+begin_src nix :exports code
# Remove the # comments from the packages section
packages = forAllSystems (system: let
  pkgs = pkgsFor system;
  # ... rest of the configuration
#+end_src

Then build with:

#+begin_src sh :exports code
# Build specific package
nix build .#cli
nix build .#server

# Build all packages
nix build .#default

# Run directly
nix run .#cli -- --help
nix run .#server -- --host 0.0.0.0 --port 8080
#+end_src

* Creating NixOS Modules

The server uses separate ~--host~ and ~--port~ arguments (rather than
a combined bind address) specifically to make it easier to create NixOS module
options:

#+begin_src nix :exports code
# Example NixOS module options
options.services.my-app = {
  enable = mkEnableOption "My App";

  host = mkOption {
    type = types.str;
    default = "127.0.0.1";
    description = "Host to bind to";
  };

  port = mkOption {
    type = types.port;
    default = 3000;
    description = "Port to listen on";
  };
};

# These map directly to CLI args
ExecStart = "${cfg.package}/bin/my-app-server --host ${cfg.host} --port ${toString cfg.port}";
#+end_src

This is cleaner than trying to parse or construct a combined address string in
Nix.

* Adding Additional Dependencies

** Workspace-Level Dependencies

Add to workspace ~Cargo.toml~:

#+begin_src toml :exports code
[workspace.dependencies]
# Your new dependency
sqlx = { version = "0.7", features = ["postgres", "runtime-tokio-native-tls"] }
#+end_src

Then use in any package:

#+begin_src toml :exports code
# crates/cli/Cargo.toml
[dependencies]
sqlx = { workspace = true }
#+end_src

** Package-Specific Dependencies

If only one crate needs it:

#+begin_src toml :exports code
# crates/server/Cargo.toml
[dependencies]
specific-crate = "1.0"  # Not in workspace dependencies
#+end_src

* Forward-porting template updates

When the template is updated after your project was emitted, pull in the
relevant infrastructure changes using =rust-template.json= at the project
root.  =docs/compliance.org= owns the authoritative process; the summary
below shows which of the two paths applies to your repo.

** Diff-based — when =template_sync_hashes= is non-empty

Take the last entry in =template_sync_hashes= as your base and diff the
template repository:

#+begin_src sh :exports code
git diff <last-hash>..HEAD
#+end_src

Every changed file is a candidate to port.  Skip project-name substitutions
and business logic; apply infrastructure changes to your project.

** Fresh-emission — when =template_sync_hashes= is empty or the file is absent

Your project predates template provenance, so there is no commit to diff
from.  Emit a fresh project from the current template into a temporary
directory and compare against it file by file:

#+begin_src sh :exports code
./new-project.sh \
  --name <your-project-name> \
  --output /tmp/<your-project-name>-fresh \
  --crates <same crates you use>
#+end_src

For every file or directory in the fresh emission, add it if absent from
your repo and reconcile if divergent — favouring the template except
where you have legitimate customisation (business logic, project-specific
configuration, domain-specific dependencies).

** Finish the update

After porting, append the current template HEAD to =template_sync_hashes=
so the next forward-port starts from the correct position:

#+begin_src sh :exports code
# Get the current template HEAD
git -C <path-to-template-repo> rev-parse HEAD
#+end_src

#+begin_src json :exports code
{
  "template_sync_hashes": [
    "<previous hashes unchanged>",
    "<paste the new hash here>"
  ]
}
#+end_src

Commit =rust-template.json= together with the ported changes.  See
=docs/compliance.org= for the discipline expected when applying changes
(what counts as compliance, what to leave alone, and how to verify).
