* DONE Rename "daemon" to "server"
The template uses "daemon" to describe the long-running service, but "server"
is more accurate.  "Daemon" describes /how/ a process is run (backgrounded,
detached from a terminal), whereas "server" describes /what/ it does (listens
for and responds to inputs).  The service is general-purpose — it may watch
local files, communicate over a binary protocol, or serve any number of roles.
HTTP is present only as infrastructure for health checks and observability, not
as the primary purpose.

This rename touches module names, NixOS/Darwin module files, systemd/launchd
unit names, documentation, and any references in =flake.nix= or =config.json=.

* DONE Document the server's role and scope
The documentation should clearly explain what the server template is /for/ vs.
what it /could/ be used for.  The server is a general-purpose long-running
service that responds to inputs — it is not specifically an HTTP server.  HTTP
is present only as infrastructure (health checks, observability, metrics) and
should be described as such.

The server might primarily watch local files, communicate over a binary
protocol, bridge between systems, or serve any other role that calls for a
persistent process responding to stimuli.  Documentation should frame HTTP as a
supporting capability rather than the defining characteristic, so spawned
projects don't inherit a misleading "web server" identity.

This is not just a cosmetic concern — LLMs reading the project documentation
have previously misidentified the server as "just for web stuff" and stood up
their own server from scratch, bypassing the template's logging, systemd socket
activation, health checks, and other infrastructure.  Clear documentation that
the server is /the/ place to build any long-running service logic should prevent
this.

* DONE Pre-commit hook formats entire tree instead of staged files only
The =.cargo-husky/hooks/pre-commit= hook passes staged filenames as
positional arguments to =treefmt=, but treefmt ignores positional arguments
and uses its own tree walker.  This means =treefmt $STAGED_FILES= formats
every file in the repository, not just the staged ones.  Unstaged files get
reformatted on disk but are not re-staged, leaving dirty working tree
artifacts that have to be committed separately.

Fix: use =treefmt --paths= (or the appropriate flag for limiting treefmt to
specific files) so only staged files are reformatted.  If treefmt does not
support per-file invocation, consider an alternative approach such as
running the language-specific formatters directly on the staged files.

* DONE Use native CA store for TLS instead of webpki-roots
The =openidconnect= crate pulls =reqwest= with =rustls-tls=, which uses
=webpki-roots= (Mozilla's CA list baked in at compile time).  This doesn't
include private CAs, so OIDC discovery fails with
~InvalidCertificate(UnknownIssuer)~ when the issuer uses an internal CA.

Fix: add =reqwest= as a direct workspace dependency with the
=rustls-tls-native-roots= feature.  Cargo's feature unification pulls in
=rustls-native-certs= so the system CA store (e.g.
~/etc/ssl/certs/ca-certificates.crt~) is read at runtime.  The crate member
that uses =openidconnect= must also list =reqwest = { workspace = true }= in its
own =Cargo.toml= for the unification to take effect.

* DONE Use Debug format for OIDC discovery errors
The OIDC discovery error handler uses ~e.to_string()~ which flattens the error
chain to just "Request failed", losing the underlying cause (e.g.
~InvalidCertificate(UnknownIssuer)~).  Change to ~format!("{e:?}")~ so the full
error chain is preserved in logs.

* TODO Darwin services must log to /var/log/<service>/ with rotation
All darwin launchd services should write ~StandardOutPath~ and
~StandardErrorPath~ to ~/var/log/<service>/~ rather than ~/tmp/~ or
nowhere.  ~/tmp/~ is cleared on reboot and is not where service logs
belong.  Without explicit log paths, launchd silently discards all
output — making post-mortem diagnosis impossible.

** Log directory creation

The log directory must exist before launchd starts the agent.  Use an
activation script in the module's ~config~ block:

#+begin_src nix
system.activationScripts.postActivation.text = ''
  mkdir --parents /var/log/<service>
  chmod 755 /var/log/<service>
'';
#+end_src

While not included here, you will also need to ~chown~ the directory if this is
a user service.

** launchd plist

Set both ~StandardOutPath~ and ~StandardErrorPath~ in the agent's
~serviceConfig~:

#+begin_src nix
StandardOutPath = "/var/log/<service>/<service>.log";
StandardErrorPath = "/var/log/<service>/<service>.log";
#+end_src

** Log rotation via newsyslog

macOS ships ~newsyslog~ which reads drop-in configs from
~/etc/newsyslog.d/~.  Add a config via ~environment.etc~:

#+begin_src nix
environment.etc."newsyslog.d/<service>.conf".text = ''
  # logfilename                          mode count size  when flags
  /var/log/<service>/<service>.log        644  5     10240 *    NJ
'';
#+end_src

- ~N~ = no signal (the service isn't syslogd; it re-opens the file on
  next write).
- ~J~ = bzip2 compression for rotated archives.
- ~size~ is in KB (10240 = 10 MB).
- ~count~ is the number of rotated archives to keep.

Adjust ~count~ and ~size~ per service — high-volume services (e.g. Ollama
inference server) need more generous limits than one-shot helpers.

** Reference commit

See =dotfiles= commit =a223015e= (~fix(ollama): add launchd log output
with newsyslog rotation~) for a working example applied to the Ollama
darwin module.
