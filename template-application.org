#+title:     Applying the Rust Template
#+language:  en

This document walks through the process of turning this template into a new
project, covering every step from cloning through first working build.

* Prerequisites

- [[https://nixos.org/download.html][Nix]] with flakes enabled.
- Git.

* Steps

** 1. Create a new repository from this template

#+begin_src sh :results none :exports code
git clone <template-repo-url> my-new-project
cd my-new-project
rm -rf .git
git init
git add .
git commit -m "Initial commit from rust-template"
#+end_src

** 2. Rename the project

Replace ~rust-template~ (kebab-case) and ~rust_template~ (snake_case) with
your project name throughout all source and config files:

#+begin_src sh :results none :exports code
PROJECT_NAME="my-app"

find . -type f \( -name "*.toml" -o -name "*.rs" -o -name "*.org" -o -name "*.md" \) \
  -not -path "*/target/*" \
  -exec sed -i '' "s/rust-template/${PROJECT_NAME}/g" {} +

find . -type f \( -name "*.toml" -o -name "*.rs" -o -name "*.org" -o -name "*.md" \) \
  -not -path "*/target/*" \
  -exec sed -i '' "s/rust_template/${PROJECT_NAME//-/_}/g" {} +
#+end_src

** 3. Update workspace metadata in Cargo.toml

#+begin_src toml :results none :exports code
[workspace.package]
version = "0.1.0"
edition = "2021"
authors = ["Your Name <your.email@example.com>"]
license = "MIT OR Apache-2.0"
repository = "https://github.com/yourusername/your-project"
#+end_src

** 4. Update flake.nix workspaceCrates

Edit the ~workspaceCrates~ section in ~flake.nix~ to reflect your actual
crates and binary names.  This is the single source of truth for Nix package,
app, and overlay generation:

#+begin_src nix :results none :exports code
workspaceCrates = {
  cli = {
    name = "my-app-cli";
    binary = "my-app-cli";
    description = "Description of your CLI tool";
  };
  web = {
    name = "my-app-web";
    binary = "my-app-web";
    description = "Description of your web service";
  };
};
#+end_src

Remove crates you do not need (see [[*Removing a crate]] below).

** 5. Rename crate directories (optional)

#+begin_src sh :results none :exports code
mv crates/cli "crates/my-app-cli"
mv crates/web "crates/my-app-web"
mv crates/lib "crates/my-app-lib"
#+end_src

Update the ~members~ list in ~Cargo.toml~ to match.

** 6. Enter the development environment and build

#+begin_src sh :results none :exports code
nix develop
cargo build --workspace
cargo test --workspace
#+end_src

** 7. Update README.org

Replace the template description with text describing your project.  Remove
references to ~template-application.org~ that are no longer relevant.

* Adding a new workspace crate

1. Create the crate:
   #+begin_src sh :results none :exports code
   cargo new --bin crates/worker
   #+end_src

2. Add to ~[workspace] members~ in ~Cargo.toml~.

3. Set up the crate ~Cargo.toml~ to inherit workspace metadata and follow the
   staged-configuration pattern from the existing crates.

4. Add an entry to ~workspaceCrates~ in ~flake.nix~.

5. Add integration tests under ~crates/worker/tests/~.

* Removing a crate

1. Remove its entry from ~[workspace] members~ in ~Cargo.toml~.
2. Remove its entry from ~workspaceCrates~ in ~flake.nix~.
3. Delete the crate directory:
   #+begin_src sh :results none :exports code
   rm -rf crates/web
   #+end_src

* Enabling Nix package builds

The ~packages~ section in ~flake.nix~ is commented out by default.  Uncomment
it when you want ~nix build .#cli~ and friends to work.  Then:

#+begin_src sh :results none :exports code
nix build .#cli
nix run .#cli -- --help
#+end_src

* Pulling template updates into a derived project

#+begin_src sh :results none :exports code
git remote add template <template-repo-url>
git fetch template
git cherry-pick <commit-hash>   # cherry-pick specific improvements
#+end_src

Or merge selectively and resolve conflicts, preserving your customizations.

* LLM instructions

This section describes what an LLM agent should do when asked to apply this
template to a new project.

** Read the project brief first

Before touching any files, confirm the new project name and purpose with the
user (or from context).  Everything else flows from that.

** Follow the steps in order

Work through [[*Steps]] top to bottom.  Do not skip renaming or metadata updates
— downstream steps assume the names are already correct.

** What to change

- ~Cargo.toml~ :: Update ~[workspace.package]~ metadata.
- ~flake.nix~ :: Update ~description~ and ~workspaceCrates~.  Match the crate
  ~name~ and ~binary~ fields to the actual Cargo package and binary names.
- ~README.org~ :: Replace this file's content with a README that describes the
  /new/ project.  Use the structure here as a guide.  Remove the template
  application and LLM sections.
- ~crates/*/src/main.rs~ :: Replace the placeholder business logic (~run()~
  function in the CLI, route handlers in the web crate) with the actual
  application.
- ~crates/*/src/config.rs~ :: Add application-specific configuration fields to
  ~CliRaw~, ~ConfigFileRaw~, and ~Config~; add semantic error variants for any
  new validation that can fail.
- ~config.example.toml~ :: Update with any new configuration keys.

** What to keep unchanged

- ~rustfmt.toml~ — formatting rules apply to all derived projects.
- ~crates/lib/src/logging.rs~ — the logging infrastructure is reusable as-is.
- ~flake.nix~ ~devPackages~ and ~devShells~ — keep the Nix shell intact.
- ~.cargo-husky/~ — keep the pre-commit hook.
- ~ERROR_HANDLING.org~ — keep or adapt; the philosophy applies everywhere.

** What to remove

- ~GETTING_STARTED.md~ and ~CUSTOMIZING.md~ — these are template meta-docs,
  not relevant once the template is applied.
- ~template-application.org~ — same as above.
- This LLM section from ~template-application.org~ (if it was copied into a
  project's docs).

** Verification

After applying the template, run:

#+begin_src sh :results none :exports code
nix develop --command cargo build --workspace
nix develop --command cargo test --workspace
#+end_src

Both must succeed with no errors before declaring the template applied.
