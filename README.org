A Rust workspace template providing best-in-class project structure and
tooling, so new projects start with everything already wired up correctly
rather than accumulating it piecemeal.

* Usage

Clone or copy this repository, then follow [[file:template-application.org][template-application.org]] for the
full process of renaming, customizing, and activating the project.

* Features

1. Integration test suite that actually runs the compiled binary.
2. Command-line arguments, environment variables, and a configuration file all
   aggregate into a single validated application configuration.
3. CI/CD workflow for versioning and publishing:
   a. Rolling ~CHANGELOG~.
   b. Published to GitHub Releases.
   c. Published to crates.io and docs.rs.
   d. Automatic version increment.
   e. Signed-committer enforcement.
4. Logging ready for structured environments (~systemd~, JSON log pipelines).
5. Quality-of-life crates included: ~tap~, ~thiserror~.
6. ~rustfmt.toml~ enforced on commit via ~cargo-husky~ (2-space indent,
   80-column width).
7. Common web service patterns out of the box:
   a. ~/healthz~ health check endpoint.
   b. ~/metrics~ OpenMetrics/Prometheus endpoint.
   c. OpenAPI specification via ~utoipa~, served with Swagger UI.
   d. ~SIGTERM~ graceful shutdown.
8. Centralized workspace crates configuration in ~flake.nix~ — one place to
   add, remove, or rename binaries.
9. Nix flake development environment via ~flake.nix~ and ~direnv~.

* Architecture Patterns

** Workspace crates configuration

~flake.nix~ contains a single ~workspaceCrates~ attribute set that drives
all Nix package, app, and overlay generation:

#+begin_src nix
workspaceCrates = {
  cli = {
    name = "rust-template-cli";
    binary = "rust-template-cli";
    description = "CLI application";
  };
  web = {
    name = "rust-template-web";
    binary = "rust-template-web";
    description = "Web service";
  };
};
#+end_src

Adding a new binary means adding one entry here; everything else
(~nix build~, ~nix run~, overlays) picks it up automatically.

** Semantic error types

Errors carry context about /what was happening/, not just /what went wrong/.
~thiserror~ is used for ergonomic error type declaration.  See
[[file:ERROR_HANDLING.org][ERROR_HANDLING.org]] for the full philosophy and patterns.

** Output and logging streams

Stdout carries program output (tables, JSON, results).
Stderr carries all ~tracing~ diagnostic output.
The ~init_logging~ function lives in the binary crate so the binary controls
where logs go.

** Staged configuration

Configuration flows through explicit typed stages:

1. ~CliRaw~ — raw command-line arguments (all ~Option<String>~).
2. ~ConfigFileRaw~ — raw TOML file contents (all ~Option<T>~).
3. Merge and validate.
4. ~Config~ — final validated state (no ~Option~ for required fields).

Validation happens at startup, producing clear error messages before any work
begins.
